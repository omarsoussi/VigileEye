Developer Documentation — How the services work together

This file documents the current implementation of the VigileEye system in this workspace: the Auth backend (port 8000), Members Invitation backend (port 8001), Camera Management backend (port 8002), Video Streaming backend (port 8003), and the SecurityFront React app. It focuses on architecture, service boundaries, request/response contracts, domain classes (fields + types), and core use cases.

At a Glance

January 21, 2026
Frontend
React + TypeScript (CRA) with API client in src/services/api.ts. Stores JWTs in localStorage and sends Authorization: Bearer to microservices.
Backends
4 FastAPI services. Each uses a Clean Architecture-ish layering: api → application → domain → infrastructure.
Auth tokens
The Auth service issues JWT access/refresh tokens with claims: sub, email, type, exp, iat. The Members and Cameras services validate those tokens locally using the shared JWT secret/algorithm.
Ports
Auth :8000 • Members :8001 • Cameras :8002 • Streaming :8003
Key contract rule: types/enums must match across services and frontend. Example: camera types are indoor|outdoor|thermal|fisheye|ptz (backend enum) and the frontend must send exactly those values.

frontend .env

If you see 401 Unauthorized from cameras/members: verify all backends load the same JWT_SECRET_KEY / JWT_ALGORITHM.

Services and responsibilities

Auth Backend
Backend/ • :8000
User accounts + OTP + JWT + Google OAuth.
  • Registration + email verification (OTP)
  • 2FA login flow (OTP)
  • JWT access/refresh issuance + refresh
  • Password reset (OTP)
Camera Management Backend
CameraManagementBackend/ • :8002
Per-user camera CRUD + enable/disable + health heartbeat.
  • Create/update/delete cameras
  • List cameras for current user (by JWT sub)
  • Enable/disable cameras
  • Heartbeat/health metrics
Members Invitation Backend
MembersInvitationBackend/ • :8001
Invite members to access camera sets.
  • Create invitation (emails a code)
  • List sent / received invitations
  • Accept/decline invitations
  • Creates Membership on accept
Video Streaming Backend
VideoStreamingBackend/ • :8003
Real-time video streaming via WebSocket.
  • Start/stop stream sessions
  • Ingest RTSP/HLS/YouTube/MP4 streams
  • Frame extraction + JPEG encoding
  • WebSocket broadcast to clients

Inter-service collaboration (how everything works together)

The system uses a shared trust model: the Auth backend is the only issuer of JWTs. Other services validate JWTs using the same secret/algorithm. The frontend is the orchestrator: it calls all three services.

Authentication → Token Storage → Authenticated calls frontend → auth → localStorage

Frontend stores
vigileye-refresh-token

Step Caller Endpoint Input Output

2 Front POST /api/v1/auth/verify { email, otp_code } MessageResponse
3 Front POST /api/v1/auth/login { email, password } MessageResponse (2FA OTP sent)
4 Front POST /api/v1/auth/login/confirm { email, otp_code } AuthResponse (user + tokens)
5 Front GET/POST to cameras & members Authorization header Data scoped to token subject

The Camera Management service derives the owner from the JWT (sub claim) and uses it as owner_user_id.

Invitation flow (share camera access) frontend → members

A user selects camera IDs (from the Cameras service) and then creates an invitation in the Members service. The Members service emails a code to the recipient. The recipient accepts using the code, producing a Membership.

Operation Endpoint Request body Returns

List received GET /api/v1/members/invitations/received (none) InvitationResponse[]

Decline POST /api/v1/members/invitations/{id}/decline DeclineInvitationRequest MessageResponse
Resend code POST /api/v1/members/invitations/{id}/resend-code (none) MessageResponse
Video streaming flow frontend ↔ streaming (REST + WebSocket)

The Streaming service has two planes: a REST control plane (start/stop streams) and a WebSocket data plane (frame delivery). The frontend calls start, then opens a WebSocket to receive JPEG frames in real-time.

Control Plane
REST /api/v1/streams/*
Data Plane
WebSocket /ws/stream/{camera_id}
Frame Format
Base64 JPEG
Operation Endpoint Request body Returns
Start stream POST /api/v1/streams/start StartStreamRequest StreamSessionResponse
Stop stream POST /api/v1/streams/stop StopStreamRequest StreamSessionResponse
Get stream status GET /api/v1/streams/status/{camera_id} (none) StreamStatusResponse
List active streams GET /api/v1/streams/active (none) ActiveStreamsResponse
Connect to stream (WS) WS /ws/stream/{camera_id}?token=... — { type: "frame", data: base64, ... }

Application layer (use cases: inputs → outputs)

Use cases are the unit of business behavior. The API layer calls use cases, use cases call repositories, and repositories persist.

Use case Input Output What it does

ConfirmLoginUseCase ConfirmLoginInput(email, otp_code) ConfirmLoginOutput(user_id, email, username, access_token, refresh_token, token_type, message) Validates OTP, issues JWT tokens.

GoogleOAuthUseCase GoogleAuthInput / callback code GoogleAuthOutput / token pair OAuth login and token issuance.
